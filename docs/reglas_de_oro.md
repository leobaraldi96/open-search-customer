# REGLAS DE ORO - DESARROLLO BUSCA CLIENTES

1. **NO ELIMINAR FUNCIONALIDADES EXISTENTES**: A menos que se pida explícitamente, nunca quitar botones, filtros, columnas o lógica de negocio que ya esté operando.
2. **RESILIENCIA DE ESTADOS**: Si un lead cae en un estado desconocido (ej: `N/A`, `null`), la UI debe seguir permitiendo acciones básicas como **AUDITAR** o **RE-AUDITAR**. No bloquees al usuario por inconsistencias en la data.
3. **PRIORIDAD DE RE-AUDITAR**: El botón de auditoría es el motor del sistema. Solo debe ocultarse cuando el proceso está ACTIVAMENTE en curso (`estado === 'investigando'`).
4. **VERIFICACIÓN EXHAUSTIVA**: Antes de dar una tarea por finalizada, DEBO abrir el navegador, probar los botones de copiado, verificar que los contadores del Dashboard sean coherentes y que los estados se actualicen en tiempo real. NO CONFIAR en la teoría; probar en la práctica.
5. **COPIADO ROBUSTO**: Usar siempre la función `secureCopy` en lugar de `navigator.clipboard` para asegurar compatibilidad en entornos locales sin HTTPS.
6. **SINCRONIZACIÓN BACK-FRONT**: Los estados finales (`analizado`, `incompleto`) deben forzarse en el backend tras el scraping para asegurar métricas coherentes en el Dashboard.
7. **STACK TECNOLÓGICO (MYSQL)**: El CRM utiliza **MySQL (XAMPP)** como base de datos, no SQLite. Es CRÍTICO manejar la serialización de campos JSON con `ensure_dict` y `ast.literal_eval` para evitar errores de mapeo en el motor de prompts.
8. **TONO ARGENTINO FEDERAL**: Al generar prompts o textos para el usuario, usar siempre **"español argentino normal"** (evitando la etiqueta 'rioplatense'). Debe sonar natural, profesional y cercano (ej: 'podés', 'chequear', 'querés'), respetando la identidad del usuario (Córdoba).
9. **UX DE ALTO RENDIMIENTO (ESTACIÓN DE COMANDO)**: Los modales de análisis deben tener un ancho de **1100px** para evitar fatiga visual. El "Prompt Maestro" y el "Resultado de Gemini" deben ser **textareas** visibles y editables. **IMPORTANTE**: Eliminar cualquier control o botón que no sea esencial para el flujo de Auditoría Manual Asistida (Command Station) para evitar ruido visual y confusión operativa.
10. **MEMORIA Y DOCUMENTACIÓN PERMANENTE**: Para no perder el hilo de la evolución del producto, toda mejora significativa DEBE quedar registrada en los archivos de documentación del repositorio (`README.md`, `docs/changelog.md` y estas `reglas_de_oro.md`).
11. **PRIORIDAD MANUAL SIEMPRE**: El dato ingresado por el usuario en el sidebar o editado en el modal SIEMPRE tiene prioridad sobre el scraping automático. El orquestador tiene prohibido sobreescribir campos de contacto (Email, Tel y Dir) si ya poseen contenido previo.
12. **ANTI-CACHÉ EN MODALES (v4.5.6)**: Toda petición `GET` que recupere datos para un modal de edición/análisis DEBE incluir un `cache-buster` (`?t=timestamp`) en la URL para evitar mostrar datos viejos tras un guardado exitoso en entornos locales (XAMPP).
13. **ATOMICIDAD DE GUARDADO (GLOBAL SAVE v4.6.0)**: El botón "GUARDAR TODO" debe persistir todos los cambios editables (Ficha + Pitch) en una sola operación atómica. No fragmentes la experiencia de guardado; el usuario debe poder editar campos técnicos y el análisis de Gemini antes de confirmar la persistencia global del lead.
14. **CABECERA FIJA Y NAVEGACIÓN (UX v4.6.1)**: En los modales de auditoría, las acciones críticas (Guardar, Cerrar) y la navegación por pestañas deben permanecer fijas en la parte superior. Esto permite habilitar un área de scroll interna infinita para la documentación técnica sin perder la disponibilidad de comandos.
15. **CONTROL CRM CENTRALIZADO (v4.6.1)**: Los cambios de estado del CRM (Contactado, Respondido, Ganado) deben realizarse desde la cabecera fija del modal mediante un selector de estado único, eliminando la redundancia de botones al final de los reportes.
16. **PROFUNDIDAD TÉCNICA (RAW v4.7.7)**: Los datos técnicos en el modal de auditoría deben ser lo más granulares posible (Performance, SEO, UX, WHOIS). SE PROHÍBE ocultar métricas recolectadas por el scraper para "simplificar" la vista; el usuario requiere la radiografía completa (items totales, metatags, botones pequeños, etc.) para su diagnóstico profesional.
17. **FORMATO DE DATOS HUMANIZADO (v4.7.7)**: Toda fecha o código técnico proveniente de APIs externas o bases de datos (ej: ISO Dates de WHOIS) DEBE ser formateado a un estándar humano (DD/MM/AAAA) antes de mostrarse en la UI para mantener la estética premium de la herramienta.
18. **CRUCE DE VERDAD (v4.9.5)**: Ante una contradicción entre los Metadatos del scraping (SEO) y la Realidad Visual (Navegación), **la visión manda**. Si el sitio está en mantenimiento o es una placa de login, se prohíbe alucinar catálogos o servicios basados en la Meta Description. El diagnóstico debe ser honesto: "Sitio Pausado".
19. **PERFIL DE SOCIO ESTRATÉGICO (v4.9.5)**: El tono de los mensajes debe ser siempre constructivo. Se prohíbe atacar a la competencia o ser agresivo con los errores del cliente. Leo se posiciona como el aliado que detecta una "fuga de valor" y propone taparla para que el negocio rinda más.
20. **INTELIGENCIA UX INVISIBLE**: El sistema no debe sonar académico. Aunque el motor utiliza heurísticas de Krug y Nielsen, el output para el cliente debe ser en lenguaje humano: "fricción al comprar", "traba en el formulario", "muro de texto difícil de escanear".
